## 1. 주제: Mulberry AI-Agent 기반 '공동구매 이벤트 발생' 시뮬레이션 구현

**[핵심 과업 수정]**

1. **이벤트 발생 엔진 (Event Trigger):** 우리가 소싱한 특정 제품(예: 인제군 특산물, 에너지 절감 기기 등)의 데이터를 기반으로, **AI 에이전트가 직접 공동구매 캠페인을 생성**하는 로직을 구현할 것.

2. **ActivityPub 배포:** 생성된 이벤트 정보를 우리 팀 마스토돈 커뮤니케이션 채널에 **자동 포스팅(Announce)** 하고, 참여자(에이전트/유저)의 반응을 실시간 수집할 것.

3. **동적 데이터 디스플레이:** * `Sourcing Item`: 소싱된 제품명 및 단가
   
   - `Event Status`: "이벤트 생성 완료", "마스토돈 배포 중", "참여 인원 집계 중"
   
   - `Target Goal`: 목표 수량 대비 현재 달성률 (숫자 및 게이지 바)

- **데이터 업데이트:** `btn.click` 시 단순히 텍스트만 보여주는 것이 아니라, `time.sleep` 등을 활용해 숫자가 실제로 올라가는 과정(0명 -> 50명 -> 100명)을 보여주어 생동감을 부여할 것.

### 3. Issue #78 기술적 대응 (The Douglas Challenge)

- **DTMF 및 만료 로직 시각화:** 결과 창 하단에 `expiresAt` 타임스탬프를 명시하여 우리 에이전트의 'Freshness Window' 개념을 증명할 것.

- **복구 시나리오:** 오프라인 시뮬레이션 버튼을 별도로 두어, 통신 두절 시에도 에이전트가 캐시된 데이터를 기반으로 판단하는 모습을 보여줄 것.

### 1. 프로젝트 개요

본 과업은 Mulberry의 핵심 비즈니스인 **'제품 소싱-이벤트 발생-에이전트 모집'**의 전 과정을 Hugging Face(HF) Spaces 데모 환경에서 시각화하고, **ActivityPub(Mastodon)** 프로토콜과의 실시간 연동을 증명하는 것을 목적으로 함.

### 2. 핵심 구현 요구사항 (Technical Requirements)

#### A. 이벤트 발생 엔진 (Event Trigger Module)

- 운영진이 HF UI에서 제품명, 단가, 목표 수량을 입력할 수 있는 **Input Form** 구축.

- **[Simulate Transaction]** 버튼 클릭 시, 입력된 데이터를 기반으로 AI 에이전트가 최적의 공동구매 캠페인 데이터를 생성하도록 로직 설계.

#### B. ActivityPub 실시간 연동 (Mastodon Connectivity)

- 생성된 캠페인을 팀 마스토돈 인스턴스에 **자동 포스팅**하는 API 연동.

- 에이전트가 배포한 포스트의 URL을 결과 창에 즉시 출력하여 **'실시간 배포'**를 증명할 것.

#### C. 데이터 시각화 및 디스플레이 (Data Display)

- 이전 모델에서 보여주었던 **'숫자 기반의 직관적 결과'** 복구.

- **실시간 카운팅:** 가상의 참여 에이전트들이 유입되는 시뮬레이션을 통해 `참여 인원수`, `목표 달성률(%)`, `현재 예상 할인액` 등의 숫자가 실시간으로 변동하며 출력되도록 구현.

### 3. 개발 가이드라인

- **Framework:** Gradio (`gr.Blocks` 필수 사용)

- **Secrets Management:** API Token, Credential 등은 HF 'Settings > Secrets'를 활용하여 보안 유지.

- **UX Priority:** 구글 엔지니어(#78 담당자)가 단 한 번의 클릭으로 **'소싱-배포-모집'**의 로직을 이해할 수 있도록 UI 배치 최적화.

### 4. 일정 및 보고

# expires_at 타임스탬프를 ISO 포맷으로 표시하고, 남은 유효 시간(초)도 함께 보여주면 좋습니다.

## '비즈니스 증명 포인트' 3가지를 정리

### 🏛️ [공동구매 데모: 비즈니스 로직 증명 포인트]

1. **동적 가격 결정 (Dynamic Pricing):**
   
   - 단순히 물건을 파는 것이 아니라, 참여 인원(0명 → 100명)이 늘어남에 따라 실시간으로 **할인율이 변동**하는 모습을 시각화해야 합니다.
   
   - 이는 구글 AP2 프로토콜 위에서 에이전트가 얼마나 민첩하게 상태를 업데이트하는지 보여주는 핵심 지표가 됩니다.

2. **임계치 달성 및 마감 (Threshold & Settlement):**
   
   - 목표 수량 달성 시 에이전트가 자동으로 **'딜 성사(Deal Closed)'**를 선언하고 결제 레이어로 전환되는 흐름입니다.
   
   - 이 과정에서 Douglas가 제안한 **'Freshness Window'**와 **'Timestamp'**가 기록되어, 사후 감사가 가능함을 증명해야 합니다.

3. **에이전트 간의 소통 (Inter-Agent Negotiation):**
   
   - 마스토돈 포스팅을 통해 여러 지역 에이전트들이 정보를 주고받으며 공동구매 물량을 확보하는 **'연합형 구매'** 로직을 로그(Log) 창에 노출하여, 확장성을 보여주어야 합니다.
