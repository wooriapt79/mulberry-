## 🔍 패턴 1: 이중 에이전트 구조 (전략 ↔ 기술 분리)

**발견된 패턴**: Mulberry는 단일 AI가 아닌 **역할 분리된 2개의 에이전트** 체계를 구축했습니다.

```
대표님 (Vision)
   ├── Malu — 수석 실장 (Strategy / 전략)
   └── Koda — CTO (Technology / 기술)
```

**핵심 증거**:
- `core.py`에서 `MulberryAgent` 클래스는 `name`과 `role`을 분리하여 인스턴스화
- README에서 **"Malu(수석실장) & Koda(CTO) 체제 구축 완료"** 라고 명시
- 두 에이전트 모두 동일한 `mission`: `"인제군 식품사막화 해소 및 시니어 케어"`

**구조적 함의**:
전략과 기술이 동일 미션을 공유하면서도 역할이 분리되어 있어, 향후 각 에이전트가 독립적으로 판단하고 협력하는 **멀티에이전트 아키텍처**로 확장 가능한 설계입니다.

**⚠️ 연속성 공백**: `core.py`에 두 에이전트 간 **통신/협업 로직이 없음**. Phase 1~3C에 이 부분이 구현되어 있는지 확인 필요.

## 🔍 패턴 2: Phase별 점진적 확장 구조

**발견된 패턴**: 코드량을 보면 Phase가 진행될수록 **기능 레이어가 추가**되는 구조입니다.

| Phase | 코드량 | 증가율 | 추정 역할 |
|---|---|---|---|
| Phase 1 | 5,500줄 | — | 핵심 기반 (데이터 수집/기초 유통) |
| Phase 2 | 4,200줄 | -24% | 로직 정제 또는 특정 기능 집중 |
| Phase 3 | 2,900줄 | -31% | AI 추론 레이어 (경량화 설계?) |
| Phase 3 보안 | 600줄 | — | 인증/암호화 모듈 |
| Phase 3-B | 750줄 | — | 추가 기능 확장 |
| Phase 3-C | 1,150줄 | — | 최종 통합/안정화 |

**핵심 인사이트**:
- Phase 1이 가장 크고(5,500줄), 이후 Phase들이 **점점 작아지는 역피라미드 구조**는 초기에 넓게 기반을 깔고 이후 집중·정제하는 방식임을 시사합니다.
- Phase 3 계열(3 / 3보안 / 3-B / 3-C)의 총합은 **5,400줄**로, Phase 1과 거의 동일 — 3단계가 실질적으로 가장 복잡한 레이어일 수 있습니다.

**⚠️ 연속성 공백**: Phase 간 **인터페이스/API 계약**이 문서화되어 있지 않음. 각 Phase가 어떤 입출력을 주고받는지 명세 필요.

## 🔍 패턴 3: 외부 연동의 다층 구조

**발견된 패턴**: `.env.example`에서 4개의 외부 API를 사용 — **소셜 / 검색 / AI / 공공행정** 레이어가 모두 포함되어 있습니다.

```
MASTODON_ACCESS_TOKEN   → 소셜 미디어 / 커뮤니티 연동
GOOGLE_API_KEY          → 검색 / 지도 / 번역 등
DEEPSEEK_API_KEY        → AI 추론 엔진
INJE_COUNTY_AUTH_CODE   → 인제군 공공 데이터 / 행정 연동
```

**구조적 의의**:
단순한 쇼핑 플랫폼이 아니라 **공공 행정 데이터 + AI 추론 + 소셜 네트워크**가 결합된 복합 플랫폼임을 보여줍니다. 특히 `INJE_COUNTY_AUTH_CODE`는 지자체와의 공식 연동이 이미 준비되어 있음을 시사합니다.

**⚠️ 보안 리스크**: `.env.example`이 GitHub에 올라가 있는 것은 정상이나, 실제 `.env` 파일이 포함되지 않도록 `.gitignore` 설정 확인이 필수입니다.

## 🔍 패턴 4: 문서-코드 간 단절

**발견된 패턴**: 18,100+줄의 코드가 존재한다고 문서에서 주장하지만, **현재 리포지토리에는 22줄(core.py)만 존재**합니다.

이 단절은 세 가지 중 하나를 의미합니다:

| 시나리오 | 가능성 |
|---|---|
| A. 코드가 아직 GitHub에 푸시되지 않음 | 높음 (`a.md`의 "GitHub 업로드" 체크리스트 미완료) |
| B. 코드가 다른 브랜치 또는 별도 리포에 있음 | 중간 |
| C. core.py가 전체 코드를 import하는 진입점 | 낮음 (현재 import 없음) |

**`a.md` 체크리스트가 이를 직접 확인해줍니다**:
```
- [x] 파일 생성
- [ ] GitHub 업로드   ← 미완료!
- [ ] 리포지토리 확인  ← 미완료!
```

**권고**: Phase 1~3C 소스 파일을 이 리포지토리에 통합하는 것이 **가장 시급한 Next Step**입니다.
